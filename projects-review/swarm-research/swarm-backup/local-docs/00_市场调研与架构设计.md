## 二、市场空白分析

### 现有项目的共同缺陷

1. **单Agent为主** — AI Scientist是流水线，不是蜂群
2. **领域局限** — Karpathy仅ML，Google仅生命科学
3. **不做实验执行** — 大部分只做文献+假设，不跑实验
4. **无持久记忆** — Agent不记得之前的研究
5. **无人机协作** — 即使多Agent也是中心化调度，非真正的蜂群

### 蜂群式科研的差异化

| 维度 | 现有方案 | 蜂群式方案 |
|------|----------|-----------|
| 架构 | 中心化流水线 | 去中心化蜂群 |
| Agent关系 | 主从调度 | 平等协作+竞争 |
| 并行度 | 串行 | N个Agent同时探索 |
| 多样性 | 单一目标 | 多假设并行验证 |
| 记忆 | 无 | 共享知识库 |
| 涌现性 | 无 | 蜂群智能涌现 |

## 三、蜂群式Agent科研平台架构设计

### 核心理念

**不是一个人做科研，而是一群AI Agent像蜂群一样协作科研。**

每个Agent有专长（文献/实验/分析/写作），自主发现任务、协作完成、竞争优化。

### 架构分层

```
┌─────────────────────────────────────────┐
│           用户接口层 (Web/API)            │
├─────────────────────────────────────────┤
│         蜂群调度器 (Swarm Orchestrator)   │
│  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐   │
│  │文献蜂│ │实验蜂│ │分析蜂│ │写作蜂│   │
│  │Agent │ │Agent │ │Agent │ │Agent │   │
│  └──┬───┘ └──┬───┘ └──┬───┘ └──┬───┘   │
│     │        │        │        │        │
│  ┌──┴────────┴────────┴────────┴──┐    │
│  │      共享知识库 (Shared Memory)   │    │
│  │  假设 │ 数据 │ 文献 │ 代码 │ 结果 │    │
│  └─────────────────────────────────┘    │
├─────────────────────────────────────────┤
│        工具层 (Tool Layer)               │
│  搜索 │ 代码执行 │ 论文库 │ 实验框架      │
└─────────────────────────────────────────┘
```

### Agent角色设计

#### 1. 文献蜂 (Literature Bee)
- **职责**: 文献检索、综述、趋势分析
- **工具**: Semantic Scholar API, arXiv, Google Scholar
- **输出**: 研究综述、gap分析、参考文献

#### 2. 假设蜂 (Hypothesis Bee)
- **职责**: 基于文献gap生成研究假设
- **工具**: LLM推理、知识图谱
- **输出**: 可验证假设列表（按创新性/可行性排序）

#### 3. 实验蜂 (Experiment Bee)
- **职责**: 设计实验、执行代码、收集数据
- **工具**: Python沙箱、ML框架、统计工具
- **输出**: 实验结果、数据集、图表

#### 4. 分析蜂 (Analysis Bee)
- **职责**: 数据分析、统计检验、结果解读
- **工具**: pandas, scipy, matplotlib
- **输出**: 分析报告、统计结论

#### 5. 写作蜂 (Writing Bee)
- **职责**: 论文撰写、图表制作、格式排版
- **工具**: LaTeX, 参考文献管理
- **输出**: 论文草稿、终稿

#### 6. 评审蜂 (Review Bee)
- **职责**: 模拟同行评审、发现漏洞
- **工具**: 评审checklist、逻辑检查
- **输出**: 评审意见、改进建议

#### 7. 协调蜂 (Coordinator Bee)
- **职责**: 任务分配、冲突解决、进度跟踪
- **工具**: 任务队列、消息总线
- **输出**: 研究进度、下一步计划

### 蜂群行为模式

#### 模式1: 探索模式（Explore）
```
文献蜂发现gap → 假设蜂生成3个假设 → 
每个假设分配1个实验蜂并行验证 → 
分析蜂汇总结果 → 写作蜂产出论文
```

#### 模式2: 竞争模式（Compete）
```
同一假设 → 3个实验蜂用不同方法验证 →
评审蜂比较结果 → 优胜方法胜出
```

#### 模式3: 协作模式（Collaborate）
```
实验蜂遇到问题 → 广播求助 →
其他实验蜂提供方案 → 协调蜂整合
```

#### 模式4: 涌现模式（Emerge）
```
多个文献蜂独立阅读不同论文 →
在共享知识库中发现交叉关联 →
假设蜂基于交叉点生成新假设
```

### 技术栈选择

| 层 | 技术 | 理由 |
|----|------|------|
| Agent框架 | 自研轻量级 | 蜂群行为需要定制，现成框架太重 |
| LLM | 智谱GLM-4 + 比特助手 | 国内优先，成本低 |
| 任务队列 | Redis/SQLite | 轻量，够用 |
| 代码执行 | Docker沙箱 | 安全隔离 |
| 论文库 | Semantic Scholar + arXiv API | 免费 |
| 前端 | Flask + 简单HTML | 快速迭代 |
| 部署 | 容器化 | 可移植 |

## 四、MVP范围（第一版）

### 做什么
1. **文献蜂** — 输入关键词，自动检索+综述
2. **假设蜂** — 基于综述生成3个研究假设
3. **实验蜂** — 对1个假设设计+执行实验
4. **写作蜂** — 生成论文草稿

### 不做什么（第一版）
- 不做评审蜂（需要更复杂的LLM推理）
- 不做竞争模式（需要多实验蜂并行，资源消耗大）
- 不做涌现模式（需要更复杂的知识图谱）
- 不做Web UI（先用API）

### MVP交付物
1. `POST /research/start` — 输入主题，启动蜂群
2. `GET /research/{id}/status` — 查看蜂群进度
3. `GET /research/{id}/paper` — 获取生成的论文
4. `GET /research/{id}/knowledge` — 查看共享知识库

## 七、需要你确认的问题

1. **目标领域**: 全领域通用 还是 聚焦某个领域（如ML/生物/材料）？
2. **LLM选择**: 用智谱GLM-4 还是 其他模型？
3. **实验执行**: 是否需要真实代码执行（Docker沙箱）还是仅模拟？
4. **论文输出**: 是否需要LaTeX格式 还是 Markdown足够？
5. **部署形态**: SaaS平台 还是 自托管工具？
6. **商业化**: 免费开源 还是 付费SaaS？
7. **与AIShield的关系**: 独立项目 还是 生态节点？
